# 金融大模型应用实施与优化方案

## 1. 项目定位

本项目面向 AI 应用开发岗位面试，目标是展示从领域数据治理、模型预训练、SFT、评估，到 RAG、推理服务和全栈应用的完整工程能力。应用场景聚焦金融文档问答、表格理解、实体与关系抽取、摘要生成和带引用的知识问答。

项目不以追求通用榜单排名为核心，而以以下能力为主要价值：

- 能否在有限算力下完成领域模型训练与优化；
- 能否通过可靠指标证明模型和 RAG 的实际增益；
- 能否构建可复现、可监控、可部署的 AI 应用；
- 能否解释数据、模型、检索和系统之间的权衡。

## 2. 当前基线

- Base checkpoint：`d24_final_mixdata` step 28000；
- SFT 最佳 checkpoint：step 800，validation bpb 为 `0.4783`；
- SFT 最终 checkpoint：step 1147，validation bpb 为 `0.4842`；
- 模型约 1.4B 参数，context length 为 2048；
- SFT 已完成双 A6000 训练，支持周期 checkpoint 和 resume；
- 数据已完成精确去重、来源划分、CoT 隔离和 FinQA/TAT-QA 文档级防泄漏；
- 已保留独立 validation 和封存 test；
- 已具备 FastAPI、React 和基础 RAG 实现。

后续模型实验以 step 800 为当前最佳起点，step 1147 仅作为过拟合对照。

## 3. 总体实施顺序

```text
轻量模型评估
→ SFT v2 数据与训练消融
→ 确定最终领域模型
→ RAG 检索与生成链路升级
→ 重点 RAG 评估
→ 可选偏好优化或金融 GRPO
→ 部署、压测和面试材料
```

当前阶段优先优化模型，不立即启动现有 RL。

## 4. 轻量模型评估

模型评估用于判断优化方向，不建设过大的通用 benchmark 系统。

### 4.1 对照模型

固定比较：

1. Base step 28000；
2. SFT step 800；
3. SFT step 1147；
4. 一个参数规模接近的通用 Instruct 模型。

所有模型使用相同测试样本、prompt、解码参数和输出限制。

### 4.2 精简指标

金融测试集按任务抽取约 100～300 条代表性样本，组成稳定的 `finance_eval_small`：

| 能力 | 指标 |
|---|---|
| 数值/表格问答 | Exact Match、数值容差准确率 |
| 实体与关系抽取 | JSON 合法率、Micro-F1 |
| 情感分类 | Accuracy、Macro-F1 |
| 摘要 | ROUGE-L、人工可用性评分 |
| 指令遵循 | 格式正确率、空回答率 |

通用能力仅保留 MMLU、GSM8K 和少量人工对话回归样本。验收目标是金融指标明显优于 Base，同时通用能力下降不超过约 2～3 个百分点。

开发阶段只使用 validation。封存 test 在最终方案确定后执行一次。

## 5. SFT v2 优化

### 5.1 数据优化

- 按 assistant 监督 token 数平衡任务，不再只按样本数重复；
- CoT 控制在总监督 token 的 10%～20%；
- 保留部分 SmolTalk、MMLU 和 GSM8K，降低灾难性遗忘；
- 对 JSON、数字和摘要任务分别统计长度、有效 token 和格式错误；
- 增加少量面向 RAG 的训练样本：
  - 根据上下文回答；
  - 上下文不足时拒答；
  - 输出引用编号；
  - 区分检索事实与模型推断。

### 5.2 训练策略

采用两阶段 SFT：

1. 通用与金融混合阶段，保持指令能力；
2. 低学习率金融专项阶段，强化金融格式和事实回答。

建议消融范围：

- `init_lr_frac`：0.05、0.1；
- CoT token 比例：10%、20%；
- 金融/通用监督 token 比例：70/30、80/20；
- 第二阶段训练长度：0.2～0.4 epoch。

训练工程增加：

- gradient clipping；
- 自动 early stopping；
- 按金融任务记录 validation 指标；
- 自动保存 best checkpoint；
- 固定随机种子和实验配置；
- 训练结束自动生成对比报告。

只进行少量、可解释的消融，避免为了面试项目消耗过多算力。

## 6. RL 与偏好优化决策

当前 `chat_rl.py` 只面向 GSM8K，缺少金融奖励、reference KL 和完整稳定机制，不直接用于金融模型。

只有满足以下条件才进入 RL：

- SFT v2 已稳定；
- 存在可靠、不可轻易投机的自动奖励；
- 通用能力回归测试已建立；
- 可以持续监控 KL、回答长度和奖励分布。

优先选择：

1. 有偏好对数据时使用 DPO/ORPO；
2. 数值 QA、JSON 抽取等可验证任务使用金融 GRPO；
3. RAG 引用任务可奖励答案正确率、引用命中率和无依据回答惩罚。

RL 模型必须相对 SFT best checkpoint 保留 KL 约束。

## 7. RAG 后续升级

RAG 是项目实际应用价值和面试展示的重点，在最终领域模型确定后继续优化。

### 7.1 检索链路

- PDF 和表格结构化解析；
- 文档、章节、表格分层切块；
- 中文与金融领域 embedding；
- BM25 与向量混合检索；
- cross-encoder reranker；
- query rewrite 和多查询召回；
- metadata、用户和文档权限过滤；
- 检索结果去重与上下文预算管理。

### 7.2 生成链路

- 统一 RAG prompt；
- 强制引用来源；
- 无证据时拒答；
- 表格与正文采用不同上下文模板；
- 对长文档使用分层检索或摘要；
- 保存 query、retrieval、prompt、answer 和反馈日志。

## 8. RAG 重点评估体系

构建约 100～200 条高质量 RAG 测试问题，覆盖：

- 单段事实；
- 跨段信息；
- 表格计算；
- 多文档对比；
- 无答案问题；
- 相似概念干扰；
- prompt injection 与权限边界。

### 8.1 检索指标

- Recall@1/3/5；
- MRR；
- nDCG@5；
- 表格命中率；
- 有效上下文覆盖率。

### 8.2 生成指标

- Answer Correctness；
- Faithfulness；
- 引用准确率与引用完整率；
- 无依据回答率；
- 拒答准确率；
- RAG 相对纯模型的任务成功率提升。

### 8.3 系统指标

- TTFT；
- p50/p95 响应延迟；
- tokens/s；
- 并发 1/4/8 吞吐；
- GPU 显存；
- 文档入库耗时；
- 检索与生成分阶段耗时。

### 8.4 应用价值指标

- 人工任务完成率；
- 专业人员可接受率；
- 相比手工检索节省的时间；
- 用户追问率；
- 错误答案严重度；
- 单次请求成本。

## 9. 面试交付物

最终仓库应提供：

- 架构图与端到端数据流；
- 数据卡、模型卡和防泄漏说明；
- Base/SFT/RAG 对比报告；
- SFT v2 消融表；
- RAG 检索、生成和系统指标；
- 可复现命令与环境说明；
- OpenAI-compatible API；
- Web 演示和预置问题；
- 失败案例、定位过程和优化结果；
- 训练、评估和服务日志截图。

面试演示建议按以下顺序：

1. 展示金融文档上传和解析；
2. 展示检索证据和引用；
3. 对比纯模型与 RAG 回答；
4. 展示模型和 RAG 指标看板；
5. 解释一次数据泄漏修复、一次训练过拟合判断和一次检索优化。

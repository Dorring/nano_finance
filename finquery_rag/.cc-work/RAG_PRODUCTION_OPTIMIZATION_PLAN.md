# FinQuery RAG 生产化优化与审核方案

## 1. 目标与边界

目标是在不改变 FinQuery 产品定位的前提下，把当前 RAG 从演示实现升级为可验证、可回放、可安全演进的生产基线。

核心原则：

1. 正确性、安全隔离优先于新增算法。
2. 每项检索优化必须通过固定评测集和历史请求回放证明有效。
3. 文档知识、会话上下文、用户长期偏好必须分层存储。
4. 不在缺少评测证据时引入 GraphRAG、复杂 Agent 或长期记忆。
5. 所有修改限制在 `finquery_rag/`；不得覆盖仓库中其他未提交改动。

## 2. 分阶段实施方案

### Phase 0：正确性与安全基线

范围：

- 修复向量检索参数顺序、异步调用和流式多文档检索问题。
- 所有向量和 BM25 记录使用租户作用域 ID，避免同名文件跨用户覆盖。
- 查询、删除、列表操作强制要求 `user_id`，禁止无过滤全库检索。
- 将“清空全部文档”改为仅清空当前用户数据，并补充身份认证。
- 修复 BM25 删除条件，保证按用户和文档删除。
- 上传使用安全临时文件，避免文件名路径穿越。
- 增加 CPU-safe 单元测试，覆盖单文档、多文档、流式、同名文件隔离及删除隔离。

退出条件：

- P0 测试全部通过。
- 构造两个用户上传同名文件时，写入、查询、删除互不影响。
- 任何缺失租户上下文的检索请求 fail closed。
- 不存在未 `await` 的异步 RAG 调用。

### Phase 1：文档生命周期与可观测性

范围：

- 新增文档注册表：`document_id`、`tenant_id`、原文件名、文件哈希、内容哈希、版本、状态、解析器/切分器/embedding 版本、创建更新时间。
- 建立索引状态机：`pending -> parsing -> indexing -> ready/failed`。
- 文件 SHA-256、规范化文本哈希和 chunk 哈希去重。
- 支持幂等重试、版本替换、删除同步和失败补偿。
- 记录结构化 trace：query、改写结果、过滤条件、候选列表及各阶段分数、上下文、模型/prompt/index 版本、耗时和错误。
- 对日志中的文档内容和用户问题提供脱敏/采样配置。

退出条件：

- 同一文件重复上传不会重复嵌入。
- 更新版本不会同时召回新旧内容。
- Dense 成功、Sparse 失败等部分失败可检测和重试。
- 任意回答能够通过 `trace_id` 重建当时使用的检索输入和版本。

### Phase 2：检索质量

范围：

- 评估并替换为适合中英金融文本的 embedding 模型。
- Dense 与 BM25 各召回 20–50 条，RRF 融合后执行 cross-encoder rerank。
- 引入 metadata pre-filter：公司、报告期、文档类型、币种、语言。
- 查询理解只负责有限路由：闲聊、事实查询、比较、摘要、计算、澄清、拒绝。
- 多轮追问改写为独立问题；必要时执行受控 query decomposition。
- Context builder 增加 chunk 去重、相邻块扩展、文档多样性和 token budget。
- 引入最低相关性阈值和无答案路径。

退出条件：

- 固定评测集 Recall@k、MRR/nDCG、数值准确率和引用准确率相对基线有统计明确的提升。
- p95 延迟和资源开销处于约定预算内。
- reranker、query rewrite 均可配置关闭并支持 A/B 对比。

### Phase 3：答案可靠性与金融工具

范围：

- 引用升级为 document/version/page/chunk，条件允许时保存 bbox。
- 生成后验证关键结论是否被引用上下文支持。
- 同比、环比、占比、加总和单位换算使用确定性计算工具。
- 对表格数字保留原值、单位、币种、期间和口径。
- 增加 prompt injection 检测，明确检索内容是数据而非系统指令。

退出条件：

- 数值类黄金集逐字段校验通过。
- 引用能够定位原始文档内容。
- 文档内恶意指令不能改变系统规则或访问其他租户数据。

### Phase 4：会话能力

范围：

- 短期记忆仅保存会话消息和独立问题改写所需摘要。
- 长期记忆仅在有明确产品需求时保存用户偏好，并与事实知识库隔离。
- 历史模型回答不得作为金融事实直接进入检索上下文。

不在当前范围：

- 无评测依据的 GraphRAG。
- 自主执行外部操作的通用 Agent。
- 将所有历史对话无差别写入向量库。

## 3. 评测与回放设计

### 3.1 黄金数据集

首批建立 100–300 条样本，至少覆盖：

- 精确数字、百分比、单位、币种、年度、季度。
- 表格、跨页、跨章节、跨文档问题。
- 中英文问题和同义表达。
- 同名指标不同口径。
- 摘要、比较和确定性计算。
- 无答案、模糊问题、错误前提。
- 越权查询和文档提示注入。

每条样本保存：

- 问题、允许检索的文档版本。
- 相关 chunk/page 标签。
- 参考答案或结构化关键事实。
- 预期引用。
- 类别、难度和回归原因。

### 3.2 指标

检索：

- Recall@k、MRR、nDCG。
- Context precision/recall。
- 文档和 chunk 去重率。

答案：

- Correctness、groundedness、answer relevance。
- Citation precision/recall。
- 数字、单位、币种、期间逐字段准确率。
- 无答案/拒答准确率。

系统：

- p50/p95/p99 总延迟和各阶段延迟。
- 错误率、超时率、索引失败率。
- 输入/输出 token 和单请求成本。
- 跨租户召回数量，目标恒为 0。

### 3.3 回放流程

1. 从线上 trace 采样失败、低分、人工差评和高价值请求。
2. 脱敏后加入回放集。
3. 新版本使用固定文档快照、模型和参数运行。
4. 输出 baseline/candidate 的逐项差异和聚合指标。
5. 质量门槛、延迟预算、安全测试全部通过后才能发布。

## 4. 审核方案

每个 Phase 均执行以下审核：

### A. 变更边界

- 检查 `git diff`，不得修改 `finquery_rag/` 之外文件。
- 不得删除或格式化用户已有无关修改。
- 不得提交密钥、数据库、模型、PDF、索引或构建产物。

### B. 代码正确性

- 检查异步边界、参数顺序、异常传播和资源清理。
- 检查所有 CRUD 及检索路径是否强制租户过滤。
- 检查 ID、upsert、更新、删除是否幂等。
- 检查 Dense 与 Sparse 索引是否可能失去一致性。

### C. 测试

- 运行新增单元测试和后端测试。
- 对安全隔离、失败分支、空结果、重复上传执行专项测试。
- 不能依赖 GPU、外部 LLM 或真实网络。

### D. 质量门禁

- 执行黄金集和历史请求回放。
- Candidate 不得降低安全指标。
- 质量提升必须同时报告延迟和成本变化。
- 无证据提升的复杂模块不合并。

### E. 整改闭环

- 审核问题按 P0/P1/P2 标记。
- P0/P1 必须修复并新增回归测试。
- 每轮整改后重新检查完整 diff 和测试结果。
- 只有所有退出条件满足后，该 Phase 才视为通过。

## 5. 当前执行顺序

本轮只执行 Phase 0。Phase 0 审核通过后，再向 CC 下达 Phase 1，不把文档注册表、reranker、记忆系统等高风险改动混入首轮修复。
